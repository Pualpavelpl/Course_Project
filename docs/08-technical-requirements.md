Требования:
Использование CSS-фреймворка (например, Bootstrap или любого другого фреймворка и набора UI-компонентов на ваш выбор);

Использование ORM (например, Sequelize, Prisma, TypeORM, Entity Framework — допустим любой вариант);
Использование системы полнотекстового поиска (с помощью внешней библиотеки или встроенных возможностей базы данных).


Зачем хранить изображения в облаке? Разве не проще сохранить их в базе данных в формате Base85, Base64 или чем-то подобном? На самом деле, для этого существуют специальные типы данных BLOB, так что даже «кодирование» не требуется.
Базы данных не оптимизированы для работы с файлами. В любой современной БД есть тип BLOB, но это не означает, что он работает эффективно. Обычно для быстрой работы записи или документы должны быть небольшими, а файл почти всегда означает «слишком большой объем данных».
В идеале файлы следует обрабатывать как потоки, а не как массивы байтов. В рамках небольшого проекта этим можно пренебречь, но если каждый из 1000 пользователей загрузит в память массив байтов размером 10 МБ, вы потеряете 10 ГБ только на файлах (не забывайте, что эти данные нужно передать через все серверы к базе данных). Конечно, современные БД (например, SQL Server) поддерживают потоковую передачу, но это потребует написания низкоуровневых SQL-запросов и решения нюансов, связанных с транзакциями и т. д. Поэтому для таких задач лучше использовать специализированное файловое хранилище.
Типичные файлы (изображения, аудио, видео) редко редактируются; по сути, они доступны только для чтения. При необходимости их просто перезаписывают целиком. Следовательно, вам нужно статическое хранилище с многоуровневым кэшированием. Здесь на помощь приходят облачные сервисы и CDN, обеспечивающие загрузку данных клиенту с быстрого сервера, расположенного в дата-центре рядом с конкретным пользователем (при этом кэширование работает максимально эффективно, так как это прямая задача облачного провайдера). Кроме того, можно выделить дополнительные ресурсы для загрузки данных с отдельного домена CDN. В целом, облачное файловое хранилище будет работать значительно быстрее, чем решение на базе БД или вариант с хранением файлов на веб-сервере рядом со скриптами. К тому же, оно гораздо лучше масштабируется.
На практике ресурсы баз данных (дисковое пространство, память, соединения) обходятся очень дорого. Поэтому передача файлов непосредственно из БД будет стоить гораздо дороже (вы платите за все серверы, участвующие в передаче данных; при использовании облачного хранилища расходы на передачу через ваши серверы тоже могут возникнуть, но см. пункт №3). Таким образом, выгоднее отдавать файлы из отдельного хранилища. Чем меньше файлов вы загружаете на свой сервер, тем спокойнее можете чувствовать себя по поводу безопасности
Разумеется, есть и минус: например, вы лишаетесь поддержки транзакций. Впрочем, при грамотной реализации для работы с изображениями они не особо и нужны. Достаточно выстроить операции с облачным хранилищем и базой данных так, чтобы в итоге появлялись лишь «мусорные» (лишние) файлы, а не «битые ссылки» в базе. Например: сначала загружаете изображение, а затем добавляете ссылку на него в базу данных. И наоборот: сначала удаляете запись из базы, а потом — сам файл изображения. В худшем случае вам понадобится простейший «сборщик мусора», который будет периодически удалять ненужные файлы в периоды низкой нагрузки. Это может быть обычная задача cron, хотя для учебного проекта это излишне. Да, даже если вы фронтенд-разработчик, вам нужно задумываться о таких вещах, как сборка мусора и согласованность данных, чтобы правильно выстраивать последовательность вызовов. Такова уж инженерная работа.

Существует множество сервисов: от гигантов вроде Amazon и Azure до простых и бесплатных решений. Возьмем, к примеру, Cloudinary: загрузить первое изображение через API и получить ссылку на него можно всего за 5 минут — включая время на вход через Google-аккаунт и установку нужной библиотеки (буквально 5 минут, я проверял лично). Всё предельно просто: есть примеры кода для JavaScript, Python, PHP, Java, C#, Ruby, Go и Dart.
Переводить пользовательские данные не требуется — интернационализация (i18n) затрагивает только элементы интерфейса (меню, ссылки, кнопки, подписи, всплывающие подсказки и т. д.). Данные, введенные пользователем (названия атрибутов, описания должностей и т. п.), сохраняются и отображаются в исходном виде.

Реализовывать графический интерфейс для управления категориями не нужно. Таким образом, пользователи не могут создавать категории, но могут создавать теги. Если владельцы приложения решат добавить новую категорию, они смогут внести изменения в соответствующую таблицу базы данных.

Теги создаются в момент первого использования — это стандартное поведение для тегов.
